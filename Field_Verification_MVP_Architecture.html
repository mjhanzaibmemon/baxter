1 Executive Summary

This document presents our revised technical architecture for your Field Agent Verification Platform — a system for automating property verification via mobile field agents. Unlike the initial ride-hailing approach, this architecture is optimized for asynchronous job assignment, photo validation, and multi-agent consensus reporting.

The solution leverages C# .NET 8 Lambda functions on AWS for cost-efficient serverless operations, intelligent proximity-based agent assignment, Amazon Rekognition for photo validation, and a custom report generation engine that triggers once all 3 agents complete their submissions.

This architecture is significantly simpler and more cost-effective than a real-time ride-hailing platform, while being purpose-built for your verification workflow.

2 Use Case & Business Flow

Your Business Model

You provide long-distance rental verification services where property owners (clients) need independent verification of rental properties for prospective tenants. The verification process requires physical presence, photographic evidence, and standardized assessment forms.

Complete Workflow

Step 1: Client Orders Service
• Client logs into portal
• Enters property address and required verification details
• Submits order and makes payment
• System creates job record with status: "Pending Assignment"
▼
Step 2: System Auto-Assigns 3 Agents
• Lambda function triggered by new job
• Query available agents near property location
• Apply scoring algorithm (proximity + agent rating + availability)
• Assign top 3 agents
• Send push notifications to agents via FCM
▼
Step 3: Agents Navigate to Property
• Agents open mobile app (Flutter)
• View job details and property location
• Use Amazon Location Service for navigation
• Agents work independently (no coordination required)
▼
Step 4: Agents Capture Photos (Outside Only)
• Take photos from multiple angles
• Upload to S3 via pre-signed URLs
• Photos validated by Amazon Rekognition (authentic, not screenshots)
• GPS coordinates embedded in metadata
▼
Step 5: Agents Fill Verification Form
• Structured form with property condition questions
• Multiple choice + text fields
• Required fields validation
• Submit button → saves to Aurora database
▼
Step 6: Once All 3 Agents Submit → Generate Report
• EventBridge detects completion (3/3 submissions)
• Lambda function triggers report generation
• Aggregates data from all 3 agents
• Detects discrepancies (if agents disagree)
• Generates PDF report + structured JSON for portal widget
• Stores in S3, updates client portal
▼
Step 7: Client Views Report
• Client receives notification (email + in-app)
• Logs into portal
• Views interactive report widget populated with verification data
• Can download PDF version if needed

Key Architectural Implications

Asynchronous by nature: Jobs can take hours or days to complete. No need for real-time WebSockets or persistent connections.

No driver-rider matching: Agents work independently. No coordination or "acceptance" flow required.

Photo authenticity matters: Amazon Rekognition validates that photos are authentic and not screenshots.

Consensus reporting: Having 3 agents provides validation through redundancy — discrepancies can be flagged.

5 Agent Auto-Assignment Algorithm

When a new job is created, the system must automatically select and assign the 3 best agents. The algorithm balances proximity, agent quality, and availability.

Assignment Logic Flow

1. Trigger: New Job Created
EventBridge rule detects new job → triggers AssignAgents Lambda function
Input: job_id, property_location (lat, lng)
▼
2. Query Candidate Agents
SQL: SELECT * FROM agents WHERE status = 'active' AND current_zone IN (<nearby zones>)
Use PostGIS extension for distance calculation: ST_Distance(agent_location, property_location)
Filter: Only agents within 50km of property
▼
3. Calculate Score for Each Agent
Score = (Proximity × 0.5) + (Rating × 0.3) + (Availability × 0.2)
• Proximity: closer = higher score (inverse of distance)
• Rating: agent's average review rating (0-5)
• Availability: 1.0 if <3 active jobs, 0.5 if 3-5 jobs, 0 if >5 jobs
▼
4. Select Top 3 Agents
Sort by score descending → Take top 3
Insert into job_assignments table: (job_id, agent_id, assigned_at, status = 'assigned')
▼
5. Notify Agents
Send push notification via FCM: "New job assigned: [Property Address]"
Update agent's active job count

Scoring Algorithm (Pseudocode)

C# — Assignment Algorithm public class AgentAssignmentService { public async Task<List<int>> AssignAgents(int jobId, Location propertyLocation) { // 1. Get candidate agents within 50km var candidates = await _db.Agents .Where(a => a.Status == AgentStatus.Active) .Where(a => ST.Distance(a.CurrentLocation, propertyLocation) < 50000) // meters .ToListAsync(); // 2. Calculate score for each agent var scored = candidates.Select(agent => new { Agent = agent, Score = CalculateScore(agent, propertyLocation) }) .OrderByDescending(x => x.Score) .Take(3) .ToList(); // 3. Assign top 3 foreach (var item in scored) { await _db.JobAssignments.AddAsync(new JobAssignment { JobId = jobId, AgentId = item.Agent.Id, AssignedAt = DateTime.UtcNow, Status = AssignmentStatus.Assigned }); // Send notification await _notificationService.SendPushAsync( item.Agent.FcmToken, "New Job Assigned", $"Property: {propertyLocation.Address}" ); } await _db.SaveChangesAsync(); return scored.Select(x => x.Agent.Id).ToList(); } private double CalculateScore(Agent agent, Location propertyLocation) { // Proximity score (closer = higher, max 50km) var distanceKm = ST.Distance(agent.CurrentLocation, propertyLocation) / 1000.0; var proximityScore = Math.Max(0, 1 - (distanceKm / 50.0)); // 0km=1.0, 50km=0 // Rating score (0-5 stars normalized to 0-1) var ratingScore = agent.AverageRating / 5.0; // Availability score var availabilityScore = agent.ActiveJobCount switch { < 3 => 1.0, < 6 => 0.5, _ => 0.0 }; // Weighted sum return (proximityScore * 0.5) + (ratingScore * 0.3) + (availabilityScore * 0.2); } }

Future Enhancements

Machine Learning: After collecting historical data, train a model to predict which agents complete jobs fastest with highest quality.

Time-of-day preferences: Some agents may prefer morning vs. evening jobs.

Language matching: For multi-region, match agent language to client language.

6 Photo Validation & Storage

Photo authenticity is critical. Agents must submit genuine photos taken on-site, not screenshots or stock images.

Photo Upload Flow

1. Agent Takes Photo
Flutter app uses camera plugin → captures image
GPS coordinates embedded in EXIF metadata (if permissions granted)
Photo stored temporarily in app storage
▼
2. Request Pre-Signed S3 URL
App calls Lambda: POST /api/submissions/{job_id}/upload-url
Lambda generates S3 pre-signed URL (valid 10 minutes)
Returns URL to app
▼
3. Upload Directly to S3
App uploads photo directly to S3 using pre-signed URL
Bypasses Lambda (no file size limits)
S3 bucket: property-photos-{env}/ with versioning enabled
▼
4. S3 Event Triggers Validation Lambda
S3 sends ObjectCreated event to EventBridge
Triggers PhotoValidation Lambda function
▼
5. Amazon Rekognition Validation
DetectLabels API → Confirm outdoor scene (building, house, etc.)
DetectModerationLabels → Check for inappropriate content
Optional: DetectText → Extract visible text (street signs, house numbers)
Optional: Compare GPS from EXIF vs. property location
▼
6. Store Validation Result
Update submissions table: photo_url, validation_status, detected_labels
If validation fails → notify agent to retake photo

Rekognition Validation Code

C# — Photo Validation Lambda public class PhotoValidationHandler { private readonly IAmazonRekognition _rekognition; public async Task<ValidationResult> ValidatePhoto(string s3Bucket, string s3Key) { // 1. Detect labels (what's in the photo?) var detectLabelsRequest = new DetectLabelsRequest { Image = new Image { S3Object = new S3Object { Bucket = s3Bucket, Name = s3Key } }, MaxLabels = 10, MinConfidence = 75F }; var labelsResponse = await _rekognition.DetectLabelsAsync(detectLabelsRequest); // Check for outdoor building-related labels var validLabels = new[] { "Building", "House", "Architecture", "Apartment", "Residence" }; var hasValidContent = labelsResponse.Labels .Any(l => validLabels.Contains(l.Name, StringComparer.OrdinalIgnoreCase)); // 2. Detect moderation labels (inappropriate content) var moderationRequest = new DetectModerationLabelsRequest { Image = new Image { S3Object = new S3Object { Bucket = s3Bucket, Name = s3Key } }, MinConfidence = 60F }; var moderationResponse = await _rekognition.DetectModerationLabelsAsync(moderationRequest); var hasInappropriateContent = moderationResponse.ModerationLabels.Any(); // 3. Validation result return new ValidationResult { IsValid = hasValidContent && !hasInappropriateContent, DetectedLabels = labelsResponse.Labels.Select(l => l.Name).ToList(), Confidence = labelsResponse.Labels.FirstOrDefault()?.Confidence ?? 0, FailureReason = !hasValidContent ? "Not a building photo" : hasInappropriateContent ? "Inappropriate content" : null }; } }

Advanced: Screenshot Detection

To detect if a photo is a screenshot (vs. real camera capture), check EXIF metadata for camera model. Screenshots typically lack camera-specific EXIF data. Additionally, Rekognition can detect UI elements (buttons, menus) that indicate a screenshot.

7 Report Generation System

Once all 3 agents submit their forms and photos, the system automatically generates a consolidated report.

Report Generation Trigger

1. Agent Submits Form
Agent fills form → Submit button → POST /api/submissions
Lambda saves submission to database
Publishes event: "SubmissionCompleted" to EventBridge
▼
2. Check Completion Status
Lambda queries: SELECT COUNT(*) FROM submissions WHERE job_id = ?
If count == 3 → All agents submitted → trigger report generation
Publish event: "JobCompleted" with job_id
▼
3. Generate Report Lambda
Triggered by "JobCompleted" event
Fetches all 3 submissions + photos from database/S3
Runs aggregation & discrepancy detection logic
Generates PDF using a library (e.g., QuestPDF for .NET)
Generates JSON for portal widget
Uploads to S3: reports/{job_id}/report.pdf
▼
4. Notify Client
Update job status: "Completed"
Send email via SES: "Your verification report is ready"
Send push notification to client app
Client can now view report in portal

Report Data Structure

SectionData SourceLogic
Property Info Job table Address, client name, verification date
Agent Details Agents table Names, IDs, submission timestamps for all 3 agents
Photos S3 URLs All photos from all 3 agents (grouped by agent)
Form Responses Submissions table Show each agent's answers side-by-side for comparison
Consensus Score Calculated % agreement across all form questions. Flag discrepancies.
Discrepancies Calculated Highlight questions where agents gave different answers

Discrepancy Detection Logic

C# — Report Generation public class ReportGenerator { public async Task<Report> GenerateReport(int jobId) { var job = await _db.Jobs.Include(j => j.Submissions).FirstAsync(j => j.Id == jobId); var submissions = job.Submissions.ToList(); // Aggregate form responses var report = new Report { JobId = jobId, PropertyAddress = job.Address }; var questions = submissions.First().FormData.Keys; // Assume all have same questions foreach (var question in questions) { var answers = submissions.Select(s => s.FormData[question]).ToList(); var uniqueAnswers = answers.Distinct().ToList(); report.Questions.Add(new QuestionSummary { Question = question, Answers = answers, ConsensusReached = uniqueAnswers.Count == 1, MostCommonAnswer = answers.GroupBy(a => a) .OrderByDescending(g => g.Count()) .First().Key }); } // Calculate consensus percentage report.ConsensusPercentage = (report.Questions.Count(q => q.ConsensusReached) * 100.0) / report.Questions.Count; // Generate PDF var pdfBytes = GeneratePdf(report); await UploadToS3($"reports/{jobId}/report.pdf", pdfBytes); return report; } }

Report Format Options

PDF: Use QuestPDF (open-source .NET library) to generate professional PDFs with charts, tables, and images.

JSON: For client portal widget, return structured JSON with all report data. Portal renders it dynamically.

Email: Send basic HTML summary via SES with link to full report.

10 CI/CD Pipeline for C# Lambda Functions

Your team requested specific guidance on CI/CD for C# AWS Lambda functions. Here's our recommended approach using GitHub Actions + AWS SAM + AWS CDK.

Pipeline Architecture

Stage 1: Build & Test (Runs on every push)
• Checkout code
• Install .NET 8 SDK
• Restore NuGet packages
• Build solution (Release mode)
• Run unit tests (xUnit)
• Generate code coverage report
• Static analysis (SonarQube optional)
▼
Stage 2: Deploy to Staging (Auto on 'develop' branch)
• SAM build (packages Lambda functions)
• SAM deploy to staging AWS account
• Run integration tests against staging
• Run smoke tests (health checks)
▼ Manual Approval ▼
Stage 3: Deploy to Production (On 'main', requires approval)
• CDK deploy infrastructure changes
• SAM deploy with alias (Blue/Green deployment)
• Gradual rollout: 10% → 50% → 100%
• Monitor CloudWatch alarms
• Auto-rollback if errors spike
• Tag release in GitHub

Pro Tips for C# Lambda CI/CD

1. Use ARM64 (Graviton2): 20% cost savings vs x86 with same performance.

2. Cache .NET dependencies: Speeds up builds. Use `actions/cache` in GitHub Actions.

3. Separate stacks: Keep infrastructure (CDK) and code (SAM) in separate stacks for faster deployments.

4. Lambda Aliases + Versions: Enable gradual rollout and instant rollback.
