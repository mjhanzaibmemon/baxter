1 Project Overview

The Baxter Data Pipeline is a production-grade automated data processing system that eliminates manual data entry for logistics operations. It monitors a Microsoft 365 mailbox, extracts Excel attachments, parses shipment and RMA data, stores it in PostgreSQL, and visualizes insights through real-time Grafana dashboards — fully automated, 24/7.

4 Data Flow Process

STEP 1 — Email Reception
Excel files arrive as attachments to monitored Microsoft 365 mailbox
↓
STEP 2 — Scheduled Polling (every N minutes)
APScheduler triggers Graph API call to fetch unread emails with attachments
↓
STEP 3 — Filtering
Filter by allowed senders and filename keywords (MS_Kargo, CS_RMA)
↓
STEP 4 — SHA-256 Deduplication Check
Hash compared against processed_attachments table — duplicates skipped, alert sent
↓
STEP 5 — Excel Parsing (streaming, read_only=True)
openpyxl iterates rows, maps columns, validates data types
↓
STEP 6 — Bulk Insert to PostgreSQL
execute_values() batch insert with ON CONFLICT upsert handling
↓
STEP 7 — Email Alert Sent
HTML success or error notification sent via Microsoft Graph API to ALERT_EMAIL
↓
STEP 8 — Grafana Dashboard Auto-Refresh
Dashboards query PostgreSQL directly — new data visible within refresh interval

6 Core Features

📧 Email Integration

Automatic monitoring of Microsoft 365 mailbox using Graph API. Filters by sender whitelist and filename keywords.

📊 Excel Parsing

Support for MS_Kargo and CS_RMA formats with automatic column mapping and type validation.

🗄️ PostgreSQL Storage

Structured schema for shipments, RMA credits, claims — with UNIQUE constraints for data integrity.

7 Production Enhancements

The following enterprise-grade features were implemented to ensure production reliability, performance, and operational visibility:

  • ✓

    SHA-256 File Deduplication

    Cryptographic hash comparison prevents duplicate file processing. Each file's hash stored and verified before any insert operation.

  • ✓

    Database Connection Pooling

    ThreadedConnectionPool (1–5 connections) prevents connection exhaustion under load and improves query performance.

  • ✓

    Bulk Insert with Upsert (ON CONFLICT)

    psycopg2.execute_values() batch inserts with conflict-resolution — 10x faster than row-by-row, handles re-runs safely.

  • ✓

    Memory-Efficient Excel Parsing

    Streaming mode (read_only=True + iter_rows) prevents memory overflow for large Excel files with 100K+ rows.

  • ✓

    APScheduler Integration

    Production-grade scheduler replacing naive time.sleep() loops. Supports missed job recovery, graceful shutdown, and configurable intervals.

  • ✓

    Retry Logic with Exponential Backoff

    Automatic retry for all Graph API and database operations. Handles transient network or service failures without crashing.

  • ✓

    HTML Email Alert System

    Automated notifications for success, processing errors, and duplicate file detection — sent via Microsoft Graph API.

  • ✓

    Docker Health Checks

    PostgreSQL and Grafana containers have health checks for auto-restart on failure and proper service dependency ordering.

  • ✓

    Security Hardening

    PostgreSQL bound to localhost only, .env secrets management, .gitignore configuration, no credentials in source code.

  • ✓

    AWS EC2 + EBS Persistent Deployment

    Production deployment on AWS with EBS-backed persistent storage, fstab auto-mount, and systemd service for auto-start on reboot.

  • ✓

    Comprehensive Documentation

    Azure AD setup guide, AWS deployment guide, and this full technical documentation for handover and future maintenance.

8 Deployment Architecture (AWS)

AWS EC2 Instance

  • Ubuntu 22.04 LTS
  • t2.micro or t3.small recommended
  • Docker + Docker Compose installed
  • systemd service for auto-start on reboot

EBS Volume

  • 5–50 GB gp3 volume
  • Mounted at /data
  • fstab entry for persistence across reboots
  • Postgres data + Grafana config stored on EBS

⚠️ Important Notes

• Email sender must be in the ALLOWED_SENDERS whitelist to be processed

• Duplicate files (same content sent again) are detected and skipped automatically — you will be notified

• PostgreSQL port 5432 is bound to localhost only — not accessible from outside the server

• All data is persisted on EBS — safe across EC2 stop/start cycles
